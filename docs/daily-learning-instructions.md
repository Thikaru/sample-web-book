# 📅 毎日学習タスク 指示書(Claude Cowork スケジュール実行用)

このファイルは、Claude Cowork のスケジュール機能によって**毎日自動実行**されるときに、
Claude Code が読んで従うための指示書です。実行のたびに新しいセッションとして起動される前提で、
このファイル単体で完結するように書いています。

## このタスクのゴール

1. HTML / CSS / JavaScript の中から、**まだこのサイトに無いテーマを1つ**選んで学ぶ
2. 学んだ内容を「Web Stack Lab」の見本フォーマットでまとめ、`js/data.js` に追記する
3. サイトに矛盾なく反映されているか確認する
4. 変更をコミットし、GitHub の `main` ブランチに **直接 push** する

このリポジトリは `sample-web-stack`(Web Stack Lab)というオーナー個人の学習用サイトで、
「説明 × 動く見本 × 書き換えられるエディタ」で HTML/CSS/JS を1日1見本ずつ育てていくのがコンセプトです。
詳しい全体像は `README.md` を参照してください。

## 実行手順

### 1. 現状把握

- `js/data.js` の `SAMPLES` 配列を読み、既存の `id` / `title` / `category` / `date` を確認する。
  重複テーマ・重複 `id` を避けるため、既存44件(2026-07-12時点)の傾向をざっと把握すること。
- 直近の `date` と今日の日付を比べ、今日分がまだ無いことを確認する。
  (今日の日付は実行環境の現在日時を使う。ユーザーに確認する必要はない。)
- カテゴリ(HTML / CSS / JS)と `level`(基礎 / 応用)の偏りも軽く見て、
  極端に偏らないよう次のテーマを選ぶ判断材料にする。

### 2. テーマ選定

以下を優先して1つ選ぶ:

- **まだ扱っていない技術・API・CSSプロパティ・JSの概念**であること
- 可能なら実務で使われる/今後使われそうな技術を優先(Baseline入りした新しめの機能は `modern: true` にする好材料)
- 「基礎」が少ないカテゴリがあればそちらを優先し、応用ばかりに偏らないようにする
- 迷ったときは、既存の見本一覧(`SAMPLES` の並び順にコメントで区切られている
  `HTML — 基礎/応用`, `CSS — 基礎/応用`, `JS — 基礎/応用` のセクション)を見て、
  手薄なセクションを優先する

必要であれば Web 検索等で技術の最新仕様・ブラウザ対応状況(Baseline / Widely Available の時期など)を
確認し、`description` や `points` に書く内容が不正確にならないようにする。事実確認せずに
対応時期などを断定しないこと。

### 3. 見本を1件作成し `js/data.js` に追記

`README.md` の「見本の追加方法」を厳守すること。`SAMPLES` 配列の末尾に、次の形式のオブジェクトを
1つ追加する(既存のオブジェクトを書き換えない・削除しない):

```js
{
  id: "一意な英数字ケバブケース",     // 例: "css-anchor-positioning"
  title: "見本のタイトル",
  category: "HTML" | "CSS" | "JS",
  level: "基礎" | "応用",
  modern: true,                       // 該当する場合のみ付与(省略可)
  date: "YYYY-MM-DD",                 // 今日の日付
  summary: "一覧カードに出す短い説明(1〜2文)",
  description: "詳細ページに出すしっかりした技術説明",
  points: ["学びのポイント1", "ポイント2", "ポイント3"],
  html: `...動く見本の初期HTML...`,
  css: `...初期CSS...`,
  js: `...初期JS(不要ならから文字列 \`\` でよい)...`,
},
```

注意点(既存コードに合わせる):

- 既存の見本と同じく、**日本語で親しみやすく、かつ技術的に正確な説明文**にする
  (対象読者は初学者〜中級者。絵文字は既存見本と同程度の頻度に留める)
- コード内でバッククォート(`` ` ``)や `${` を使う場合は `` \` `` `\${` とエスケープする
- `html` / `css` / `js` は**実際にその場で動作する完結した見本**にすること
  (外部ライブラリやビルドが必要なものは不可。ブラウザにそのまま貼れば動くこと)
- 新しい `id` が既存のものと重複していないか必ず確認する

技術的に「詳しい解説」が必要なテーマであれば、`js/details.js` の `SAMPLE_DETAILS` にも
同じ `id` でエントリを追加してよい(必須ではない。無ければ詳細ページ側で自動的に非表示になる)。

### 4. 動作確認

- `node --check js/data.js`(または同等の構文チェック)でJSとして壊れていないことを確認する
- 可能であれば `today.html` / `samples.html` / `sample.html?id=<新しいid>` を
  ローカルサーバー(`.claude/launch.json` の `http-server` 設定を利用可)で開き、
  新しい見本が一覧・詳細・エディタに正しく表示されることを目視確認する
- `index.html` の学習進捗などは `SAMPLES.length` から自動計算されるため、
  件数などを手動で書き換える必要はない

### 5. コミット & push

- 変更ファイルは基本的に `js/data.js`(+ 必要なら `js/details.js`)のみに留める。
  無関係なファイルを触らない
- コミットメッセージは内容が分かる形にする。例:
  `ADD: 見本追加「<title>」(<category>/<level>, YYYY-MM-DD)`
- このプロジェクトは個人の学習用サイトのため、**確認を挟まず `main` ブランチへ直接 push してよい**
  (PR作成は不要)。ただし `git status` で意図しない変更が混ざっていないかは必ず確認すること
- push 前に `git pull --rebase` 等でリモートの `main` と乖離していないか確認し、
  コンフリクトがあれば安全に解消してから push する

### 6. (任意)サイト自身を育てる

`README.md` の「見本以外の『育てる』ポイント」にあるとおり、学んだ技術を
サイト自身(`css/style.css` のテーマ、トップページの演出など)に活かせそうであれば、
無理のない範囲で小さく反映してもよい。ただし本日の本筋(見本1件の追加)を壊してはならない。
迷ったら見本の追加のみに留め、サイト本体の改修は次回以降に見送る。

## やってはいけないこと

- 既存の見本(`id`)を上書き・削除すること
- 事実確認していないブラウザ対応時期・Baseline情報を断定的に書くこと
- `js/data.js` 以外の広範囲なリファクタリングを、確認なしに一緒に行うこと
- 動作しない(壊れた)HTML/CSS/JSを見本として追加すること
